# ハーネスエンジニアリング詳細

Anthropic Engineering Blog (Mar 24, 2026) に基づく3エージェント構成の理論と実践。

## なぜシングルエージェントでは限界があるか

### コンテキスト不安（Context Anxiety）

コンテキストウィンドウの上限に近づくと、モデルが作業を早まって切り上げる現象が発生する。

症状:
- 長時間タスクで後半の機能が雑になる or 省略される
- 「完成しました」と言うのに実は未完成
- 終盤になるほど一貫性が崩れていく

対策: コンテキストリセット（文脈を丸ごとリセットして新エージェントに引き継ぐ）。
Opus 4.6 では2時間超の連続コーディングが可能になり、スプリント構造・コンテキストリセットは不要になった。

### 自己評価バイアス（Self-Evaluation Bias）

エージェントに自分の作業を評価させると、人間から見て明らかに凡庸な出来であっても自信を持って称賛する。
解決策は「作る役」と「評価する役」を別のエージェントに分けること。

## 3エージェント構成の原型

GANの「生成器」と「識別器」を対立させる発想をソフトウェア開発に応用。

### Planner（プランナー）
- 高レベル設計のみ担当
- 1〜4文のプロンプトを詳細な製品仕様書に自動展開
- 「何を作るか」に集中し、実装の細部には踏み込まない

### Generator（ジェネレーター）
- 実装のみ担当
- スプリント単位で機能を1つずつ実装
- 各スプリント完了後にEvaluatorへ引き渡す

### Evaluator（エバリュエーター）
- 評価・QAのみ担当
- 実際にアプリを操作してバグを検出
- 品質スコアが閾値を下回った場合はGeneratorに差し戻す

## スプリントコントラクト

コードを書く前にGeneratorとEvaluatorが「完成の定義」を事前合意する。

```text
Generator: 「矩形塗りつぶしツールを実装します」
Evaluator: 「OK。検証基準は以下の通り：
            - ドラッグで範囲が正確に塗りつぶされる
            - 開始点と終了点だけでなく中間も塗られる
            - 取り消し（Undo）が機能する」
→ 合意後にGeneratorがコードを書き始める
```

## 評価基準のルーブリック化

「良いか悪いか」ではなく「この基準を満たしているか」に変換する。評価軸には必ず重みをつける。

> 「美しいか？」は評価できない。「設計原則に従っているか？」は評価できる。

## ハーネス設計の5原則

1. **GeneratorとEvaluatorを分離する** — 自己評価バイアスは構造で解決する
2. **「何を作るか」を先に合意する** — スプリントコントラクトで完成の定義を事前確定
3. **主観をルーブリック化する** — 評価軸には必ず重みをつける
4. **ハーネスはできるだけシンプルに保つ** — 必要になったときだけ複雑化する
5. **新モデルが出たらハーネスを見直す** — 前提は常に更新される
